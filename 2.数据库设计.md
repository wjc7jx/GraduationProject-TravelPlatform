### 1. 数据库设计思路

本系统的数据库设计基于 **MySQL 8.0**，统一采用 `InnoDB` 存储引擎，字符集选用 `utf8mb4`（以支持Emoji表情及多语言字符）。设计过程中遵循以下原则：

1. **适度规范化（遵循第三范式，3NF）**：消除数据冗余，如将“位置信息”单独成表，避免在每个内容记录中重复存储经纬度和详细地址，保证数据的一致性。
2. **实用性与扩展性优先**：考虑到旅行内容类型丰富（图、文、音、轨迹各存在极大的结构差异），在 `contents` 表中将差异化数据收敛至核心的 `content_data` JSON 字段，大幅减少表字段稀疏与利用率低的问题，同时满足各种媒体额外属性的灵活扩展。
3. **软删除机制**：核心数据表（如项目、日记等）采取软删除（增加 `is_deleted` 或 `status` 字段），避免用户误删导致的旅行记忆彻底丢失，同时便于进行历史数据归档。
4. **统一命名规范**：表名采用小写英文复数形式（如 `users`），字段名采用下划线蛇形命名法（如 `user_id`），保证命名具备清晰的业务自指性。

### 2. 表结构总览

本系统核心涵盖以下 5 张数据表：

| 表名 | 逻辑名称 | 核心作用 |
| --- | --- | --- |
| `users` | 用户信息表 | 存储用户基础信息、小程序授权信息及账号标识。 |
| `projects` | 旅行项目表 | 作为旅行的时间轴容器，管理每次旅行的全局信息（起止时间、封面等）。 |
| `contents` | 多媒体内容表 | 聚合各个旅行项目下的照片、日记、音频及轨迹等具体内容记录。 |
| `locations` | 位置信息表 | 存储解析或关联的 GPS 坐标与结构化地址，供地图绘点使用。 |
| `permissions` | 权限规则表 | 管理旅行项目或具体内容的隐私可见性（私密、好友、公开）。 |

### 4. 表间关系说明

系统的表之间存在紧密的逻辑关联（实体关系：ER 结构）：

*   **一对多关系：`users` (1) ↔ `projects` (N)**
    一名用户可创建、维护多个独立的旅行项目（如"川西环线"、"周末露营"）。
*   **一对多关系：`projects` (1) ↔ `contents` (N)**
    一个具体的旅行项目是由多个多媒体节点（照片、日记、声音、路线等）组成的，它们根据内容对象中的 `record_time` 构建出项目的时间轴。
*   **多对一关系：`contents` (N) ↔ `locations` (1)**
    不同的照片或日记可以标记为同一个位置。通过外键解耦 `location_id`，避免位置信息的重复并提高地图数据的查询与聚合效率。
*   **多态关联关系：`permissions` ↔ `projects` 或 `contents`**
    利用业务分离设计的 `target_type` 和 `target_id`，权限表不仅可以约束整个项目的可见度，也可以细粒度到把项目内的某一篇带"家庭住址"的日记设置为私密。

### 5. 索引设计建议

为保障大量多媒体内容检索及项目拉取时的系统性能，提供以下索引设计：

1. **唯一索引 (Unique Index)**：
   * `users.openid`：用于小程序快捷自动登录，提高校验速度，防重复绑定。
2. **普通索引 (Normal Index)**：
   * `projects.user_id`：用户打开应用首页，优先筛选自身所有的旅行项目，属于极高频查询。
   * `contents.project_id`：打开某旅行项目进入详情、构建全局时间轴时的前置条件。
3. **复合索引 (Composite Index)**：
   * `contents(project_id, record_time)`：针对“左侧时间轴、右侧地图联动”的核心场景。根据项目拉取内容时往往默认按记录时间升序/降序排列，复合索引可避免二次 Filesort，直接命中索引出库。

### 6. 设计合理性说明

1. **微信业务适配**：`users` 表以 `openid` 为核心，而非强制绑定手机号作为最初鉴权。此设计符合当前微信小程序“静默登录/快捷登录”的生态趋势，最大程度降低用户初始使用门槛。
2. **高稀疏字典表 JSON 整合**：针对旅行中的照片、日记、音频等差异化介质，系统摒弃了拆分多张附表或是预设大量冗余空字段的做法。而是抽取核心排序特征（如 `record_time` 和 `location_id`）为独立字段，将差异化属性（如照片的宽高、相机的EXIF、富文本内容、音频的时长参数）统一整合至 `content_data` 核心 JSON 字段。此举显著提高了单表的列级利用率，充分利用 MySQL 8.0 优异的 JSON 特性，既满足了多态数据存取的无限扩展，又能保障基础时空轴线的查询性能。
3. **坐标类型精度**：`locations` 表经纬度选用 `DECIMAL(10,7)`。7位小数对应的实际地面精度约 1.1 厘米，可完全覆盖用户 GPS 轨迹描绘及照片 EXIF 高精解析的需求，且不会存在浮点丢失现象计算地图偏倚。
