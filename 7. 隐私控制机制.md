# 旅行项目管理系统 - 隐私控制机制（实施方案）

本文档基于当前项目代码现状给出可落地的隐私控制实施方案，目标是满足“私密/好友可见/公开”三级可见性、导出过滤、地理位置脱敏，并保证后端与小程序行为一致。

---

## 一、背景与现状

### 1. 当前已具备能力

1. 已有权限规则表 `permissions`，支持：
   - `target_type`：`project` / `content`
   - `target_id`
   - `visibility`：`1私密`、`2好友可见`、`3公开`
   - `white_list`：白名单用户ID数组
2. 后端导出服务已具备初步可见性过滤参数：`visibility_scope`、`viewer_user_id`。
3. 鉴权已使用 JWT 中间件，具备基础访问控制能力。

### 2. 当前主要缺口

1. 内容列表、时间地图等主查询链路尚未统一使用同一套隐私判定。
2. 权限判定逻辑散落在导出模块，未形成可复用的统一服务。
3. 小程序端暂无“项目默认可见性/内容覆盖可见性”配置入口。
4. 地理位置“模糊处理”尚无统一策略与实现位置。

---

## 二、实施目标

### 1. 业务目标

1. 支持项目级默认可见性与内容级覆盖可见性。
2. 支持好友关系访问（好友可见），白名单作为可选扩展。
3. 导出 PDF/HTML、时间地图、内容列表遵循统一隐私规则。
4. 对位置数据进行按权限分级脱敏。

### 2. 工程目标

1. 所有隐私判定统一收敛到一个服务层，避免重复实现。
2. 兼容当前数据库结构，优先少改动、低风险。
3. 提供明确的接口契约，便于小程序快速接入。
4. 具备可验证的测试清单与验收标准。

---

## 三、总体设计

### 1. 设计原则

1. **单一判定源**：权限判定只在后端服务层执行，前端不自行推断。
2. **默认继承 + 局部覆盖**：内容级权限未设置时继承项目级权限。
3. **显示与导出一致**：用户在页面可见的数据与导出结果遵循同一规则。
4. **最小暴露**：默认最严格（私密），显式授权后再放开。

### 2. 可见性模型

- `1 私密`：仅资源所有者可见。
- `2 好友可见`：所有者 + 已建立好友关系用户可见（可叠加白名单扩展）。
- `3 公开`：可被任意已登录用户查看（后续可扩展匿名公开）。

### 3. 权限优先级

对某条内容 `content` 的最终可见性，采用以下优先级：

1. 若存在 `content` 级权限规则，使用该规则。
2. 否则继承所属 `project` 级权限规则。
3. 若项目规则也不存在，使用系统默认规则：`私密`。

可抽象为：

$$
finalRule(content)=contentRule\;?\;contentRule:(projectRule\;?\;projectRule:defaultPrivate)
$$

## 四、后端实现方案

### 1. 新增统一隐私服务层（核心）

建议新增 `src/services/privacyService.js`，提供以下能力：

1. `getProjectRule(projectId)`：读取项目规则，不存在返回默认私密。
2. `getContentRules(contentIds)`：批量读取内容规则，减少N+1查询。
3. `resolveContentRule(content)`：计算内容最终规则（内容覆盖项目）。
4. `canView(rule, ownerUserId, viewerUserId)`：判断是否可见。
5. `filterViewableContents(contents, viewerUserId)`：过滤可见内容。
6. `sanitizeLocation(item, viewerLevel)`：按可见级别脱敏坐标。

### 2. 规则读写接口

新增路由建议：

1. `GET /api/projects/:id/privacy`
2. `PUT /api/projects/:id/privacy`
3. `GET /api/projects/:projectId/contents/:contentId/privacy`
4. `PUT /api/projects/:projectId/contents/:contentId/privacy`
5. （可选）`PUT /api/projects/:projectId/contents/privacy/batch`

请求体建议：

```json
{
  "visibility": 1,
  "white_list": [10001, 10002]
}
```

约束建议：

1. `visibility=2` 时，优先按好友关系判定；`white_list` 可为空，作为扩展判定。
2. `visibility!=2` 时，`white_list` 自动置空。
3. 仅资源所有者可修改权限。

### 3. 查询链路改造点

将以下服务统一接入 `privacyService`：

1. 内容列表接口：`contentService.listContents`
2. 时间地图接口：`projectService.getTimelineMapOverview`
3. 项目导出接口：`exportService.buildExportData`
4. 未来新增“公开浏览/分享页”接口

统一策略：

1. 先查业务数据（项目/内容/位置）。
2. 批量查权限规则。
3. 做可见性过滤。
4. 对可见数据做位置脱敏。

### 4. 导出策略统一

保留现有参数能力：

1. `visibility_scope=all|public|share`
2. `viewer_user_id`
3. `include_content_ids` / `exclude_content_ids`

并增加约束：

1. `all` 仅允许所有者使用。
2. `share` 必须提供 `viewer_user_id` 或由 token 识别。
3. 导出前再次调用统一判定层，不直接相信调用方参数。

### 5. 地理位置脱敏策略

建议采用分级精度策略（经纬度小数位）：

1. 所有者查看：6位（米级）
2. 好友可见：4位（约10米级）
3. 公开：3位（约100米级）
4. 私密且非所有者：不返回经纬度，仅返回模糊地址（如城市）或空值

可近似表示误差：

$$
error\approx 111000\times 10^{-n}\; (m)
$$

其中 $n$ 为保留小数位数。

### 6. 安全与审计

1. 所有隐私设置操作记录审计日志（操作者、目标、前后值、时间）。
2. 对外分享链接增加有效期与签名（后续增强项）。
3. 禁止通过前端字段绕过权限过滤（后端兜底）。

## 五、小程序接入方案

### 1. 页面改造

1. 项目编辑页：增加“默认可见性”设置。
2. 内容编辑页：增加“继承项目/单独设置”开关。
3. 内容编辑页：当单独设置时可选择`私密/好友可见/公开`，好友可见由系统按好友关系自动判定。
4. 导出页：增加导出范围说明（全部/公开/分享可见）。

### 2. 交互与提示

1. 选择“公开”时提示“位置信息将按公开策略脱敏显示”。
2. 选择“好友可见”时提示“将按好友关系自动判定，无需输入ID”。
3. 内容继承项目策略时，UI明确展示“当前实际可见性”。

### 3. 前后端职责边界

1. 前端负责配置与展示。
2. 后端负责最终判定与过滤。
3. 前端不可依赖本地状态判断“是否可见”。

## 六、数据库与兼容策略

### 1. 推荐方案（当前项目优先）

继续以 `permissions` 表作为唯一权限来源，不在 `contents` 表新增 `visibility` 字段，理由：

1. 已有模型与数据基础，改造成本低。
2. 支持 project/content 多态目标，扩展性更好。
3. 避免字段重复导致一致性问题。

### 2. 答辩兼容方案（可选）

若论文或评审必须强调 `contents.visibility`：

1. 在 `contents` 增加冗余字段 `visibility_cache`（可选名）。
2. 写入权限时双写（permissions + cache）。
3. 读取时以 `permissions` 为准，cache仅用于统计与检索。

## 七、测试与验收

### 1. 功能测试场景

1. 私密内容：所有者可见，非所有者不可见。
2. 好友可见：好友用户可见，非好友不可见（可选白名单用户同样可见）。
3. 内容覆盖项目：项目公开 + 内容私密时，该内容仍私密。
4. 项目私密 + 内容公开：该内容可公开（按内容覆盖策略）。
5. 导出HTML/PDF与页面展示结果一致。
6. 地图点位在不同可见级别下脱敏精度正确。

### 2. 安全测试场景

1. 篡改 `viewer_user_id` 参数不能越权。
2. 缺失 token 或无效 token 访问导出接口应拒绝。
3. 白名单注入非法类型（字符串、对象）应校验失败。

### 3. 性能测试场景

1. 单项目 1000 条内容列表查询与过滤耗时可接受。
2. 导出时批量读取权限，避免逐条查询。

## 九、风险与应对

1. **风险：规则散落导致行为不一致**
   - 应对：所有查询统一走 `privacyService`。
2. **风险：好友关系滥用（陌生人误加）**
   - 应对：仅接受分享卡片落地邀请，后续可扩展二次确认或拉黑机制。
3. **风险：公开内容泄露精确住址**
   - 应对：公开强制坐标降精度 + 地址脱敏。
4. **风险：导出链路绕过权限**
   - 应对：导出前二次判定，不信任前端参数。

---

## 十、结论

本方案以“统一判定层 + 项目默认继承 + 内容覆盖 + 分级脱敏”为核心，在不推翻现有结构的前提下，能够较快完成隐私控制系统建设，并满足毕业设计中的隐私控制验收要求。后续可平滑扩展到分享链接、好友关系模型与审计追踪。
