
# 模块 2：内容记录与多媒体管理 答辩清单
## 1. 模块职责概述
本模块是“行程时间轴”应用的数据核心，主要负责图文笔记、相片、语音和轨迹等多媒体内容的增删改查。它通过安全的直传凭证（Qiniu Token）机制支持云端大规模并发上传，并对写入的富文本、外部 URL 进行严格清洗，确保页面展示的安全且兼顾时间轴或视图动态排序的需求。

## 2. 核心代码链路说明
+ **前端（小程序端）**：
    - **核心逻辑**：主要交互集中在 `TripTimeline/miniprogram/pages/editor/`（内容发布页）与 `TripTimeline/miniprogram/components/audio-recorder/`（录音组件）。
    - **网络发包**：
        1. **上传前置**：调用 `POST /upload/qiniu-token` 根据多媒体类型 (`purpose`) 申请七牛云安全直传凭证，或调用本地 `POST /upload/photo` 及 `/upload/trajectory` 获取 EXIF 和轨迹元数据解析。
        2. **内容拉取或落表**：拿到多媒体直链后，携带富文本、GPS对象等信息，调用 `POST /projects/:projectId/contents` 将内容存入系统的 `contents` 表结构中。
+ **后端（Node.js 端）**：
    - **Controller (**`src/controllers/contentController.js`** & **`uploadController.js`**)**：负责项目下内容的基础 CRUD 路由绑定；`uploadController` 还承接多媒体服务器端代理转切和智能解析功能（EXIF或轨迹解析）。
    - **Service (**`src/services/contentService.js`** & **`qiniuUploadToken.js`**)**：
        * `qiniuUploadToken.js`：对外部调用方做 `purpose` 参数策略限制（针对图片、音频、GPS文件限定不同大小与扩展名）；并根据凭证生成 `publicUrl`。
        * `contentService.js`：除了 CRUD 外，调用 `normalizeContentData` 函数负责将非预期的脏数据做拦截过滤。
    - **Model (**`src/models/content.js`**)**：使用 `content_type` 强定义素材性质 (`photo`, `note`, `audio`)，将动态属性全部存放到无模式的 `content_data` (JSON) 中，提升多态扩展性。

## 4. 亮点与技术难点实现解析
1. **大规模图片/音频并发直传（高可用设计）**：彻底移除了经过系统 Node.js 内存转发大文件的瓶颈（除了须服务端做 EXIF 萃取的接口外）。通过 `qiniuUploadToken.js` 实现客户端直连云存储节点，业务数据库只保留文本 URL 指针；大大避免了高并发打爆 Node 单线程的内存和网络带宽。
2. **结构化强类型与动态 JSON 结合设计**：在 `contents` 表中对于基础检索强依赖项（如 `record_time`, `location_id`, `project_id`, `content_type`）做了显式列定义加上联合索引优化查询范围时间轴；而针对如图片多尺寸URL数组、富文本内容、录音时长等零散的业务展示字段，统包置于 `content_data` 的 `JSON` 字段中保存，平衡了检索效率和横向扩展的灵活性。
3. **严格的数据卫生隔离防护**：在写入落表前，利用 `sanitizeRichText` 工具做了 DOM 级别的清洗，防止可能遭受的注入式跨站脚本攻击 (XSS) 等风险。

## 5. 答辩导师高频 Q&A 预测
### Q1: 照片、语音、富文本这几种不一样的数据结构，你是怎么在数据库里存进去并和项目关联的？
> **答辩话术**：我采用了传统的 **核心索引列** 搭配 **JSON聚合列** 来构建 `contents` 表。诸如所属的 `project_id`、发生的 `record_time`（用于时间轴排序关键）、以及经纬度 `location_id`，我设成了独立列并加了数据库索引以提高时间轴渲染和分页的检索效率；而那些多态的数据（如文字的 HTML，图片的 URL 和 EXIF 数组，音频的名称时长）则抽象放在一套 `content_data` （JSON 类型列）里，这样后续拓展其它记录类型，不需要频繁 alter 数据库表结构。
>

### Q2: 为什么你要加一套独立的 `qiniuUploadToken` 服务去拿票据，而不让小程序直接传图片给后端，后端存表再存云上去？
> **答辩话术**：因为在典型的旅游时间轴记录场景下，一次性发原图组或者大量的语音会有极大的带宽和 IO 开销。如果将大文件全通过 Node.js 转发，非常容易导致内存耗尽从而让其他轻量查询接口宕机。利用 `qiniu-token` 机制可以让端直传对象服务商的分布式 CDN 上，我的服务器只承载最终极小字节的 URL 地址入库，实现了计算、存储、网络架构的解耦和高并发承载力。
>

### Q3: 你的 EXIF 数据解析、传图片、又存正文，整个接口链路会不会太长？遇到失败或者回退怎么办？
> **答辩话术**：在系统设计时我做了解耦处理。将资源“上传获取空间地址”和最终关联发布的“内容提交”作为前后台分离的操作。上传图片失败或卡顿不会导致数据表脏写；只有用户在客户端真正点击 `发布` 时，才会批量整理所有预传拿到好的 URL 和富文本，进行一次原子操作的 JSON 入库（此时即便断网，内容草稿也仍保存在前端），这保证了应用层的数据一致性与事务整洁度。而且我对 `purpose` 参数在发 token 前做了类型跟预期文件大小拦截，从源头去除了部分失败重试。
>
